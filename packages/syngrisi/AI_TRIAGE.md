# AI Triage (AI-сортировка диффов) — Beta

> Статус: Beta (реализовано) — фича доступна, по умолчанию выключена и включается отдельно для каждого проекта.
> Связано: [AI_FEATURES.md](./AI_FEATURES.md), [RCA.md](./RCA.md)

## Обзор

**AI Triage** автоматически классифицирует каждый упавший чек с помощью vision-LLM
(VLM), чтобы ревьюеры перестали тонуть в сырой «красноте» диффов. Для каждого
упавшего чека модель возвращает вердикт, уровень уверенности и одну фразу-причину.
Ревьюер видит уже отсортированный список («3 реальных регрессии, 9 шума») вместо
десятков одинаково красных чеков.

Это **одна фича с двумя режимами действия**, а не две фичи:

- **Triage (триаж)** = получить вердикт. Работает всегда (когда включено).
- **Auto-accept (автопринятие)** = опциональная политика **на уровне проекта (App)**,
  которая по высокоуверенному вердикту автоматически принимает безопасные чеки, оставляя
  людям только спорное. У каждого проекта своя политика.

Auto-accept — это просто агрессивный режим того же вердикта: ни второй модели, ни
второго прохода, ни второго эндпоинта.

### Какую проблему решает

~90% времени ревью уходит на отсев шума (антиалиасинг, динамический контент, тайминг
рендера) и на повторы (одно изменение CSS красит 50 чеков). Triage снимает отсев шума;
будущая фича группировки снимет повторы.

## Вердикты

| Вердикт | Значение | По умолчанию можно автопринимать |
| :--- | :--- | :--- |
| `intended_change` | Реальное, намеренное изменение UI → ожидается новый baseline | Да (опционально) |
| `likely_bug` | Неожиданная регрессия, нужен взгляд человека | Нет |
| `noise` | Антиалиасинг / динамический контент / дрожание рендера — не реальное изменение | Да (опционально) |
| `uncertain` | Модель недостаточно уверена для классификации | Нет |

> **Важно:** `likely_bug` — это *рекомендация/подсветка* от AI, а не бэкенд-статус
> `markedAs: 'bug'`. Triage сам по себе никогда не выставляет `markedAs`, кроме случая
> явной политики авто-принятия (которая ставит `accepted`). В текущем UI доступны
> только **Accept** и **Remove** — Triage остаётся в рамках этой модели.

## Модель данных

В модель Check добавляется одно поле. Вердикт вычисляется **один раз** и читается и
UI, и политикой авто-принятия, и аудитом.

```typescript
// src/server/models/Check.model.ts
triage?: {
  verdict: 'intended_change' | 'likely_bug' | 'noise' | 'uncertain';
  confidence: number;        // 0..10 (целое; 10 = максимальная уверенность)
  reason: string;            // одна человекочитаемая фраза-объяснение
  model: string;             // напр. "gpt-4o", "gemini-2.0-flash", "claude-sonnet-4-6"
  at: Date;
  autoAccepted?: boolean;    // true, если чек принят политикой автоматически
}
```

Аудит получается бесплатно: автопринятый чек показывается как
`accepted by AI (conf 9.4)` — то же поле, без отдельного лога.

## Как это работает

```
check.created (вебхук/событие)
        │
        ▼
фоновый triage-job ──► VLM через настроенного провайдера
        │              (переиспользует пейлоад GET /ai/analysis/:id:
        │               base64 baseline/actual/diff + метаданные + опц. DOM-дифф)
        ▼
запись Check.triage { verdict, confidence, reason, model, at }
        │
        ▼
читаем политику проекта (App) этого чека
        │
        ▼
policy = suggest ? ничего не делать (только UI)
       : auto     ? если verdict ∈ App.triageAutoAcceptVerdicts
                       && confidence ≥ App.triageAutoAcceptThreshold (0..10)
                      → batch-accept (существующий POST /ai/batch), autoAccepted=true
                    иначе → оставить человеку
```

Triage строится на уже существующей инфраструктуре: `GET /ai/analysis/:id` уже отдаёт
base64-картинки + контекст, готовые для VLM, а `POST /ai/batch` уже умеет accept/remove.
DOM-дифф из RCA, когда есть, передаётся как дополнительный контекст для повышения
качества вердикта.

## Настройки

Настройки делятся на два уровня: **глобальные** (включение фичи и провайдер модели —
общие для инстанса) и **политика автопринятия на уровне проекта** (своя для каждого
App).

### Глобальные (на инстанс)

| Настройка | Описание | По умолчанию |
| :--- | :--- | :--- |
| `ai_triage_enabled` | Главный вкл/выкл триажа | `false` |
| `ai_triage_provider` | Активный провайдер модели (см. ниже) | — |

Хранятся как сейчас: строка в `AppSettings.model.ts`, сид в
`src/seeds/initialAppSettings.json`, с полем `env_variable`, чтобы оператор
self-hosted мог переопределить через переменные окружения (системный env > админка,
как сейчас).

### Политика автопринятия — на уровне проекта (App)

**Важно:** автопринятие настраивается отдельно для каждого проекта (App), а не
глобально. У разных проектов разная зрелость и риск-профиль: на одном можно агрессивно
автопринимать `noise`, на другом — только `suggest`. Поэтому политика — это поле(я) на
сущности App, а не глобальная настройка.

| Поле (на App) | Описание | По умолчанию |
| :--- | :--- | :--- |
| `triagePolicy` | `suggest` (только UI) или `auto` (действовать по вердикту) | `suggest` |
| `triageAutoAcceptThreshold` | Мин. уверенность (0..10) для автопринятия | `9` |
| `triageAutoAcceptVerdicts` | Какие вердикты можно автопринимать | `intended_change`, `noise` |

> Классификация (показ вердикта) отделена от автопринятия. Вердикты можно видеть
> всегда в режиме `suggest`; автопринятие — отдельный осознанный опт-ин, причём
> включается **на конкретном проекте**. Нельзя зашивать автопринятие намертво в триаж.

Хранятся в `src/server/models/App.model.ts`. Редактируются в настройках проекта; при
обработке вердикта фоновый job читает политику того App, к которому относится чек.

## Провайдеры моделей (админка)

Новый раздел **AI Providers** в **Admin → Settings** (`/admin/settings`,
`AdminSettings.tsx`) позволяет администратору настроить, какой LLM-бэкенд питает
триаж. Реализуется на существующих формах настроек (`Settings/Forms/`), по аналогии с
тем, как у SSO есть своя форма `SsoSettingsForm.tsx`.

Поддерживаемые типы провайдеров:

| Провайдер | Примечания |
| :--- | :--- |
| **OpenAI** | API-ключ + имя модели (напр. `gpt-4o`) |
| **Google Gemini** | API-ключ + имя модели (напр. `gemini-2.0-flash`) |
| **Anthropic** | API-ключ + имя модели (напр. `claude-sonnet-4-6`) |
| **OpenAI-совместимый (self-hosted)** | Base URL + опц. ключ + имя модели. Покрывает Ollama, vLLM, LM Studio, LiteLLM, Together, OpenRouter и т.д. |

Настройки на экране (на каждого провайдера):

- Выбор типа провайдера
- API-ключ (write-only, маскируется после сохранения)
- Base URL (для OpenAI-совместимых / self-hosted)
- Имя модели
- Кнопка «Test connection» → лёгкая проверка доступности провайдера (`ping`), а не полная
  классификация: для OpenAI-совместимых/Ollama это `GET /models` (заодно проверяется, что
  указанная модель есть в списке), для Anthropic — запрос на 1 токен, для Gemini — запрос
  метаданных модели. Фиксированный таймаут 15 c, поэтому UI не зависает на медленных
  локальных VLM. Показывает задержку + имя модели
- Опционально: temperature, max tokens (пусто = без лимита), request timeout (ms, пусто = 120000)

Замечание про self-hosting: поскольку self-hosted-вариант говорит на
OpenAI-совместимом формате, никакие данные Syngrisi не обязаны покидать инфраструктуру
оператора — это хорошо ложится на позиционирование self-hosted и privacy-first.

## Экраны UI

| Подфича | Экран | Где именно |
| :--- | :--- | :--- |
| Бейдж вердикта на чеке | Главная страница сравнения (`index2`) | Строка чека / заголовок `CheckDetails` — цветной чип вердикта + уверенность. **Клик по бейджу → фильтрует все чеки с таким же вердиктом в текущем ране** |
| Детали вердикта + причина | `CheckDetails` | Небольшая Triage-панель рядом с RCA (паттерн `RCAPanel.tsx`), открывается из `Toolbar` |
| Перезапуск триажа | `CheckDetails/Toolbar/Toolbar.tsx` | Действие в тулбаре (иконка), рядом с переключателем RCA |
| Группировка по вердикту | Навбар (`NavbarGroupBySelect.tsx`) | Новый пункт «Group by → AI Verdict», компонент-элемент по аналогии с `AcceptStatusItem` |
| Фильтры триажа | Навбар (`NavbarFilter.tsx`) | Фильтр по вердикту (`intended_change` / `likely_bug` / `noise` / `uncertain`), по `reason` и по диапазону `confidence` (0..10). Чекбокс **«only current run»** ограничивает фильтр текущим раном |
| Индикатор автопринятия | `CheckDetails` / статус accept | Лейбл «Accepted by AI (conf 9.4)» на существующем статусе принятия |
| Включение фичи + провайдер | Admin → Settings (+ AI Providers) | Глобальные настройки `ai_triage_*` через существующие `Settings/Forms` |
| Политика автопринятия (per-project) | Настройки проекта (App) | `triagePolicy` / порог / разрешённые вердикты — своя для каждого App |

Новых верхнеуровневых страниц не требуется: вердикты живут на существующей странице
сравнения, глобальная конфигурация — в админке, а политика автопринятия — в настройках
проекта.

### Поведение фильтров (детали)

- **Фильтр по `reason`** — текстовый/подстрочный фильтр по фразе-причине (например,
  показать всё, где причина содержит «dynamic content»).
- **Фильтр по `confidence`** — диапазон 0..10 (например, `≥ 8` — только уверенные
  вердикты).
- **Чекбокс «only current run»** — когда включён, любой из фильтров (вердикт / reason /
  confidence) применяется только к чекам текущего рана, а не ко всей выборке.
- **Клик по бейджу вердикта** — быстрый шорткат: ставит фильтр «этот вердикт» +
  «only current run», то есть мгновенно показывает все такие вердикты в текущем ране.

## User stories

### Ревьюер (режим по умолчанию `suggest`)

- **Как ревьюер**, я хочу, чтобы каждый упавший чек был помечен вердиктом (намеренное /
  баг / шум / не уверен) и одной фразой-причиной, чтобы решать с одного взгляда, не
  вглядываясь в каждый дифф.
  *Экран:* список чеков `index2` + `CheckDetails`.
- **Как ревьюер**, я хочу группировать и фильтровать чеки по AI-вердикту, по причине
  (`reason`) и по диапазону уверенности (`confidence` 0..10), чтобы сначала ревьюить все
  `likely_bug` и пропускать `noise`.
  *Экран:* группировка/фильтры в навбаре.
- **Как ревьюер**, я хочу чекбокс «only current run» у фильтров, чтобы ограничить выборку
  только текущим раном и не отвлекаться на исторические чеки.
  *Экран:* фильтры в навбаре.
- **Как ревьюер**, я хочу кликнуть по бейджу вердикта на чеке и сразу увидеть все чеки с
  таким же вердиктом в текущем ране, чтобы быстро разобрать однотипные результаты пачкой.
  *Экран:* бейдж вердикта в списке / `CheckDetails`.
- **Как ревьюер**, я хочу читать, *почему* модель так решила (и видеть контекст
  DOM-диффа, когда он есть), чтобы доверять вердикту или переопределять его.
  *Экран:* Triage-панель в `CheckDetails` (рядом с RCA).
- **Как ревьюер**, я хочу перезапускать триаж для отдельного чека после изменений,
  чтобы обновить устаревший или неверный вердикт.
  *Экран:* действие в тулбаре `CheckDetails`.
- **Как ревьюер**, я хочу, чтобы в режиме `suggest` AI никогда молча не менял чек — он
  только подсказывает, а **Accept**/**Remove** я нажимаю сам.

### Ревьюер / владелец CI (режим `auto`)

- **Как владелец CI**, я хочу настраивать политику автопринятия отдельно для каждого
  проекта, чтобы на зрелом проекте включить `auto`, а на новом оставить `suggest` — без
  влияния одного проекта на другой.
  *Экран:* настройки проекта (App).
- **Как владелец CI**, я хочу, чтобы высокоуверенные `intended_change` и `noise`
  автопринимались выше заданного мной порога (0..10), чтобы пайплайн показывал только
  то, что требует человека.
  *Экран:* настройки проекта (`triagePolicy = auto`, `triageAutoAcceptThreshold`).
- **Как ревьюер**, я хочу, чтобы автопринятые чеки были явно помечены «Accepted by AI
  (уверенность)», чтобы аудит был очевиден и я мог выборочно проверить модель.
  *Экран:* статус принятия в `CheckDetails`.
- **Как владелец CI**, я хочу ограничить автопринятие конкретными вердиктами (никогда
  `likely_bug`/`uncertain`), чтобы модель не могла сама одобрить реальную регрессию.
  *Экран:* настройки проекта (`triageAutoAcceptVerdicts`).

### Администратор

- **Как админ**, я хочу включать/выключать AI Triage глобально, чтобы контролировать,
  работает ли фича вообще.
  *Экран:* Admin → Settings.
- **Как админ**, я хочу выбирать провайдера — OpenAI, Gemini, Anthropic или
  OpenAI-совместимый self-hosted эндпоинт — и вводить его ключ/base URL/модель, чтобы
  триаж использовал бэкенд, разрешённый моей организацией.
  *Экран:* Admin → Settings → AI Providers.
- **Как админ на self-hosted**, я хочу направить триаж на собственный
  OpenAI-совместимый эндпоинт (Ollama/vLLM/и т.д.), чтобы никакие скриншоты не покидали
  мою инфраструктуру.
  *Экран:* Admin → Settings → AI Providers (Base URL).
- **Как админ**, я хочу кнопку «Test connection», которая быстро проверяет доступность
  провайдера и модели (без полной классификации), чтобы убедиться, что провайдер работает,
  до включения триажа.
  *Экран:* Admin → Settings → AI Providers.
- **Как админ**, я хочу задать лимит запросов/бюджета, чтобы ошибка конфигурации или
  огромный ран не привели к неограниченному счёту за LLM.
  *Экран:* Admin → Settings → AI Providers.
- **Как админ**, я хочу переопределять настройки провайдера через переменные окружения,
  чтобы конфигурировать секреты в CI/Docker без UI.
  *Поведение:* тот же приоритет env-над-UI, что и у существующих настроек.

## Рекомендуемая модель для разработки и тестов

Для разработки и тестирования фичи используйте **Qwen3-VL 8B**, запущенную локально
через **Ollama**. Это разумный дефолт: сильное понимание UI/скриншотов, хороший OCR
(~89.6% OCRBench), нативный multi-image (можно подать baseline + actual + diff одним
запросом) и OpenAI-совместимый эндпоинт — то есть тестовый стенд сразу совпадает с
продакшн-вариантом провайдера «OpenAI-compatible (self-hosted)».

```bash
ollama pull qwen3-vl:8b
ollama serve   # OpenAI-совместимый эндпоинт: http://localhost:11434/v1
```

Настройки провайдера для тестов:

- Тип провайдера: **OpenAI-compatible (self-hosted)**
- Base URL: `http://localhost:11434/v1`
- Имя модели: `qwen3-vl:8b`
- API-ключ: любой непустой (Ollama его игнорирует)

Замечания при тестировании:

- Подавайте модели **diff-картинку вместе с baseline/actual** — VLM плохо ловят мелкие
  попиксельные сдвиги сами, но хорошо классифицируют уже подсвеченную область.
- Включайте строгий JSON-вывод (`format: json` в Ollama) для стабильного
  `{ verdict, confidence, reason }`.
- **Калибруйте порог auto-accept на своих данных** — 8B-модель завышает confidence,
  поэтому начинайте с режима `suggest`, измеряйте точность вердиктов и только потом
  включайте `auto`.

> Требования: ~10–12 ГБ VRAM для `qwen3-vl:8b` (Q4). Для слабого железа — `qwen3-vl:4b`,
> для максимального качества на спорных кейсах — `qwen3-vl:32b`.

## План реализации (TDD)

Фича реализуется в TDD-стиле: **сначала e2e-тесты на Gherkin, затем код**. Для каждой
подфичи пишется feature-файл (падающий), и только потом — реализация, доводящая тест до
зелёного.

Порядок:

1. **Написать Gherkin feature-файлы** в `e2e/features/` (область `ai_features` или новая
   `AI_TRIAGE`), покрывающие сценарии ниже. Прогнать `npx bddgen` — тесты падают (нет
   реализации).
2. **Шаги (steps)** в `e2e/steps/` — переиспользуя существующие шаги (bootstrap БД/сервера,
   создание чеков, навигация, ассерты) по [common steps cheatsheet](./agent/guides/common_steps_cheatsheet.md).
   Провайдер модели в тестах — мок/фейк (детерминированный вердикт), чтобы тесты не
   зависели от реального LLM. Отдельный «живой» прогон против локального Qwen3-VL — вручную.
3. **Реализация** (модель данных, фоновый job, эндпоинты, UI) до зелёных тестов.
4. **Проверка стабильности**: `cd e2e && npx bddgen && npx playwright test "<feature>" --workers=3 --repeat-each=4`.

### Сценарии (черновик Gherkin)

```gherkin
Feature: AI Triage verdicts

  Scenario: Failed check gets a verdict in suggest mode
    Given a project with AI Triage enabled and policy "suggest"
    And the triage provider returns verdict "noise" with confidence 9
    When a failed check is created
    Then the check shows an AI verdict badge "noise"
    And the check is not auto-accepted

  Scenario: Auto-accept applies per project above threshold
    Given a project "Web" with policy "auto", threshold 9, verdicts "intended_change,noise"
    And a project "Mobile" with policy "suggest"
    And the triage provider returns verdict "noise" with confidence 10
    When a failed check is created in project "Web"
    And a failed check is created in project "Mobile"
    Then the check in "Web" is accepted by AI
    And the check in "Mobile" is not auto-accepted

  Scenario: Verdict below threshold is left for a human
    Given a project with policy "auto" and threshold 9
    And the triage provider returns verdict "intended_change" with confidence 7
    When a failed check is created
    Then the check is not auto-accepted
    And the check shows verdict "intended_change" with confidence 7

  Scenario: likely_bug is never auto-accepted
    Given a project with policy "auto", threshold 9, verdicts "intended_change,noise"
    And the triage provider returns verdict "likely_bug" with confidence 10
    When a failed check is created
    Then the check is not auto-accepted

Feature: AI Triage filtering

  Scenario: Click verdict badge filters same verdict in current run
    Given a run with checks having verdicts "noise", "noise", "likely_bug"
    When I click the verdict badge "noise" on a check
    Then only checks with verdict "noise" from the current run are shown

  Scenario: Filter by confidence range
    Given checks with confidence 3, 7 and 10
    When I filter by confidence "greater or equal" 8
    Then only the check with confidence 10 is shown

  Scenario: Filter by reason
    Given checks with reasons "dynamic banner" and "font changed"
    When I filter by reason "banner"
    Then only the check with reason containing "banner" is shown

  Scenario: Only current run checkbox limits the filter scope
    Given the current run and a previous run both have checks with verdict "noise"
    When I filter by verdict "noise" with "only current run" enabled
    Then only "noise" checks from the current run are shown
```

> правила селекторов (ARIA roles > Labels > Text > TestID > CSS/XPath), см.

## Вне рамок этой фичи

- **Умная группировка / дедупликация** множества чеков от одного изменения — отдельная
  фича, строится на том же вердикте + DOM-диффе.
- **Авто-генерация ignore-зон** из обнаруженных динамических областей — отдельная фича.
- **Правила триажа на естественном языке** («игнорируй изменения цвета < 5%») — возможно
  позже через систему плагинов.

## Открытые вопросы

- Где запускать triage-job: синхронно на `check.created` или в фоновом воркере с
  очередью? (Предпочтительно фон — чтобы не блокировать создание чека и батчить вызовы.)
- Контроль стоимости: бюджет на ран vs лимит на месяц vs оба.
- Как обрабатывать ошибки/таймауты провайдера — оставлять вердикт `uncertain` и
  показывать небольшое состояние «triage failed», а не блокировать ревью.
- Триажить ли по умолчанию только чеки с диффом (`hasDiff`), чтобы экономить токены.
