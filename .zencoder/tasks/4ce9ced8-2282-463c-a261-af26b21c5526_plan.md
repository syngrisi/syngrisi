# Cape plan

## Recommended scope
Сделать **minimal useful Cape** как **отдельный human-friendly CLI client** рядом с existing MCP bridge, а не встраивать human layer в [./packages/syngrisi/e2e/support/mcp/bridge-cli.ts](./packages/syngrisi/e2e/support/mcp/bridge-cli.ts).

Причина: `bridge-cli.ts` и [./packages/syngrisi/e2e/support/mcp/bridge.ts](./packages/syngrisi/e2e/support/mcp/bridge.ts) уже держат корректный STDIO JSON-RPC path; safest way не ломать его — оставить bridge entrypoint неизменным и повесить Cape сверху как отдельного клиента, который спаунит `bridge-cli.ts` и общается с ним через MCP client/stdio transport.

## Phases

### Phase 1 — Cape wrapper
Добавить REPL-style Cape CLI, который:
- спаунит `bridge-cli.ts` как child process;
- использует `Client` + `StdioClientTransport` по тому же паттерну, что уже есть в [./packages/syngrisi/e2e/support/mcp/test/utils/index.ts](./packages/syngrisi/e2e/support/mcp/test/utils/index.ts);
- маппит human-friendly команды на existing MCP tools;
- печатает только human-readable output в свой stdout;
- корректно шлет shutdown notification и чистит child process.

### Phase 2 — Cape skill
Добавить узкий `.gemini` skill по паттерну [./.gemini/skills/review-changes/SKILL.md](./.gemini/skills/review-changes/SKILL.md), который описывает Cape workflow и guardrails, но **не меняет** MCP config surface.

### Phase 3 — Tests
Добавить отдельные тесты для Cape CLI и оставить существующие bridge tests как regression net.

## Recommended human-friendly command surface
Для v1 — **simple REPL**, а не mixed-mode bridge и не rich TUI.

Recommended commands:
- `/start <sessionName> [--headed]` → `sessions_clear` + `session_start_new`
- plain line with step text → `step_execute_single`
- `/batch` → collect >=2 lines, then `step_execute_many`
- `/attach` → `attach_existing_session`
- `/clear` → `sessions_clear`
- `/tools` → list available tools
- `/exit` or `/shutdown` → send shutdown notification and exit cleanly

Why this surface:
- минимальный useful Cape subset;
- keeps one live MCP session/process;
- reuses existing step/runtime tooling without inventing external session persistence.

## Isolation from STDIO JSON-RPC path
Hard boundary: **Cape process is the human-facing client; bridge child remains pure JSON-RPC transport**.

Implementation rule:
- human input/output lives only in `cape-cli.ts`;
- `bridge-cli.ts` child `stdin/stdout` are private pipes owned by `StdioClientTransport`;
- no human-friendly messages from bridge to stdout;
- bridge logs remain file-only / stderr-only as they are today;
- do not add TTY branching into `bridge-cli.ts`, because that risks changing current MCP behavior.

## Key trade-offs
- **Chosen trade-off**: one extra client-side wrapper process in exchange for near-zero risk to current bridge transport.
- **Benefit**: maximal reuse of `bridge.ts`, `server.ts`, MCP SDK client/transport, and current tests.
- **Cost**: one more entrypoint/script to maintain.

## Non-goals
- no rewrite of `bridge.ts`/`server.ts`
- no change to MCP tool contracts
- no switch of `.gemini` configs from `bridge-cli.ts` to Cape
- no rich TUI/autocomplete/history beyond basic REPL
- no new abstraction layers around existing step/runtime infra
