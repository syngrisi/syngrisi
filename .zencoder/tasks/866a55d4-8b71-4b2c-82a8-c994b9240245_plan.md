# Practical implementation plan: Cape wrapper + skill + tests

## Recommended approach
Сделать **Cape как тонкий отдельный human-friendly CLI поверх existing MCP bridge**, а **agent skill** — как отдельный `.gemini` skill, который использует уже существующий `test_engine` MCP server.

**Не менять контракт текущего `bridge-cli.ts`**: он должен остаться pure stdio JSON-RPC entrypoint для MCP-клиентов.

## Scope

### Reuse as-is
- `packages/syngrisi/e2e/support/mcp/bridge.ts`
- `packages/syngrisi/e2e/support/mcp/server.ts`
- `packages/syngrisi/e2e/support/mcp/bridge-cli.ts`
- existing MCP tools: `session_start_new`, `attach_existing_session`, `step_execute_single`, `step_execute_many`, `sessions_clear`, shutdown notification
- existing `.gemini/antigravity/mcp_config.json` pointing to `bridge-cli.ts`

### Add / change
- **add** `packages/syngrisi/e2e/support/mcp/cape.ts`
- **add** `packages/syngrisi/e2e/support/mcp/cape-cli.ts`
- **add** `.gemini/skills/cape/SKILL.md`
- **add** `packages/syngrisi/e2e/support/mcp/test/mcp-cape-cli.spec.ts`

- **optional minimal change** `packages/syngrisi/e2e/package.json` only if нужен удобный script alias для запуска Cape CLI/tests
- **optional minimal change** `packages/syngrisi/e2e/support/mcp/test/utils/index.ts` only if окажется выгоднее переиспользовать общий launcher/helper вместо дублирования

## Human-friendly command surface
Для v1 — **простой REPL/CLI**, без TUI и без mixed stdout mode:

- `/start <sessionName> [--headed]` → `sessions_clear` + `session_start_new`
- `/attach` → `attach_existing_session`
- plain text line → `step_execute_single`
- `/batch` → собрать `>=2` шага и вызвать `step_execute_many`
- `/clear` → `sessions_clear`
- `/tools` → list tools
- `/shutdown` or `/exit` → shutdown notification + cleanup

### Why this surface
- минимально полезно для Cape browser flow
- reuse текущих MCP tools без нового runtime
- не требует менять `server.ts` или tool contracts
- легко тестируется поверх уже существующих helper patterns

## How to avoid breaking existing stdio behavior
Жесткое правило:
- `bridge-cli.ts` остается **pure JSON-RPC over stdio**
- все human-friendly prompts/output живут только в `cape-cli.ts`
- `cape-cli.ts` общается с bridge через `Client` + `StdioClientTransport`
- не добавлять дружелюбные сообщения, аргумент-парсинг или TTY-ветвление в `bridge-cli.ts`
- если нужны логи Cape CLI — печатать их в human CLI layer, не в bridge transport stdout

Итог: MCP/agent path не меняется, Cape path — отдельная обертка поверх него.

## Skill scope (minimal useful subset)
В `.gemini/skills/cape/SKILL.md` описать только полезный минимум:
- когда использовать skill
- обязательный session lifecycle: `session_start_new` или `attach_existing_session`
- правило: single action/diagnostics → `step_execute_single`
- правило: `step_execute_many` только для `>=2` шагов
- recovery path: `sessions_clear` → restart session
- graceful shutdown
- ссылки на уже существующие guide/docs вместо дублирования документации

### Important skill constraints
- не пытаться делать full-featured browser platform
- не invent new MCP tools
- не менять `.gemini/antigravity/mcp_config.json`, если текущий `test_engine` уже работает через `bridge-cli.ts`

## Phased steps

### Phase 1 — Cape wrapper shell
1. Добавить `cape.ts` с thin wrapper around MCP client lifecycle:
   - start bridge child via `StdioClientTransport`
   - create MCP client
   - helper methods: `startSession`, `attachSession`, `executeSingle`, `executeBatch`, `listTools`, `shutdown`, `cleanup`
2. Добавить `cape-cli.ts` как human entrypoint (`#!/usr/bin/env node`), который:
   - читает stdin
   - маршрутизирует slash-команды
   - plain lines отправляет как `step_execute_single`
   - корректно завершает transport/client

### Phase 2 — Cape skill
1. Добавить `.gemini/skills/cape/SKILL.md`
2. Использовать паттерн структуры из `.gemini/skills/review-changes/SKILL.md`
3. В инструкции skill явно опереться на существующий `test_engine` и current MCP tool contract

### Phase 3 — Tests
1. Добавить `test/utils/cape.ts` для запуска Cape CLI и обмена stdin/stdout/stderr в тестах
2. Добавить `mcp-cape-cli.spec.ts` с минимальным coverage:
   - happy path: `/start` → один step → `/exit`
   - batch path: `/start` → `/batch` с 2 валидными шагами
   - attach flow через recorded port
   - invalid command / invalid batch
   - graceful shutdown / cleanup
3. Existing `mcp-bridge-cli.spec.ts` не переписывать существенно; использовать как regression net для pure bridge behavior

### Phase 4 — Optional polish only if cheap
- добавить package script alias для Cape CLI smoke run
- слегка вынести общие launcher/helpers, если это уменьшает дублирование без усложнения

## Trade-offs
- **Chosen trade-off**: отдельный Cape entrypoint вместо расширения `bridge-cli.ts`
  - плюс: почти нулевой риск сломать JSON-RPC path
  - минус: +1 CLI entrypoint и немного test harness кода
- **Chosen trade-off**: skill only for minimal browser/session workflow
  - плюс: быстро и без overengineering
  - минус: не будет advanced UX/TUI/history/autocomplete в v1
- **Non-goals**:
  - переписывать `bridge.ts` / `server.ts`
  - менять MCP tool contracts
  - full-featured Cape platform
  - переносить `.gemini` MCP config с `bridge-cli.ts` на Cape wrapper
